# Architecture Principles Reference

## 1. 平台定位

平台采用“单一智能核心 + 多个独立外挂模块”的形态。

- 核心负责需求理解、Skill 选择、模块路由、跨模块编排、新模块开发与修复。
- 外挂模块只负责自己的确定性能力闭环。
- 核心本身也是独立应用，不是散落在仓库中的抽象概念。

## 2. MVP 冻结原则

MVP 目录是原型基线，不作为新平台的演进目录。

- 所有新核心能力、新模块和新共享库都放在 Modules 下。
- 如果需要复用 MVP 中的实现，应复制到 Modules 体系后继续维护。
- 不允许让 Modules 下的新模块直接依赖需要持续演进的 MVP 源文件。

## 3. 模块独立应用原则

每个模块都必须具备独立应用边界，至少包括：

- 独立入口
- 独立配置
- 独立日志
- 独立数据目录
- 独立生命周期
- 独立测试

这样核心运行时才可能对模块执行单独开发、测试、启动、停止、替换、回滚和观测。

## 4. 职责边界

### 核心负责

- 理解需求
- 读取 Skill
- 选择模块
- 编排跨模块流程
- 管理生命周期、权限、回调和观测

### 模块负责

- 接收核心调用
- 完成单一职责执行
- 向指定 replyTo 回调结果
- 提供自己的 Skill 和 Manifest

## 5. 模块分类

### adapter

接入外部系统或外部事件，例如 IM、Webhook、邮件、浏览器面板。

### executor

提供确定性执行能力，例如定时调度、OCR、文件转换、报表生成。

### manager

提供治理能力，例如配置中心、日志中心、模块控制台。

## 6. 工作区统一，运行隔离

模块可以放在同一个工作区下，便于智能体统一阅读代码与生成 Skill；但运行数据必须隔离。

模块只能写自己的：

- runtime/data
- runtime/logs
- runtime/tmp

模块之间共享信息必须通过统一消息协议完成。
