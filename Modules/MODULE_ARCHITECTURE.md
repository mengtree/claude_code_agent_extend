# 可自我迭代智能平台架构草案

## 1. 目标

本平台的目标不是让每个模块都具备智能体能力，而是在核心平台中内置统一智能体能力，由核心负责：

- 理解用户需求
- 理解和选择 Skill
- 判断是否调用已有模块
- 判断是否需要开发新模块
- 编排多个模块完成一条业务链路
- 修复、迭代、替换已有模块

这里的“核心”不是一个抽象概念，而是一个实际存在的核心模块。核心模块本身也是平台中的一个独立应用，只是它承担了统一智能体能力与统一编排职责。

各个外挂模块只负责自己的职责闭环，不默认具备智能体能力。模块是核心平台的外挂扩展，而不是独立的大脑。

本文档负责定义架构原则与边界；具体实施顺序、阶段目标、交付物与验收标准，以 Modules/IMPLEMENTATION_PLAN.md 为执行基线。

## 2. 总体原则

### 2.1 单一智能核心

全局只有一个核心智能体中心。所有需要理解、规划、分析、开发、跨模块协同的事情，都由核心智能体处理。

核心模块既是平台控制面，也是唯一默认内置智能体能力的模块级应用。也就是说，核心不是“凌驾于模块之外”的特殊代码堆，而是需要像其他模块一样被启动、部署、观测和演进的独立应用，只是在职责上位于全局中枢位置。

### 2.1.1 MVP 冻结原则

MVP 目录用于保留当前原型实现和验证基线，后续模块化平台建设不直接在 MVP 中演进。

约束如下：

- MVP 保持不动，作为对照样板和可参考实现
- 所有新模块一律在 Modules 目录下全新开发
- 如果需要复用 MVP 中的实现，应复制到 Modules 下对应模块，再在复制后的代码上继续演进
- 不允许把 Modules 的模块化改造反向混入 MVP

这个原则的目标是保留一个稳定原型基线，避免平台化重构过程中把 MVP 变成半成品过渡层。

### 2.1.2 模块独立应用原则

每个模块都必须被设计为一个独立应用，而不只是核心仓库里的一个内部类库或一组函数。

这样设计的原因不是形式上的解耦，而是为了给后续自我迭代留下真实边界。只有模块本身具备独立入口、独立运行上下文、独立配置、独立日志、独立数据目录和独立发布单元，核心智能体才可能在后续针对某个模块执行以下动作：

- 单独开发
- 单独测试
- 单独启动
- 单独停止
- 单独替换版本
- 单独回滚
- 单独观测与排障

如果模块只是核心内部的源码片段，那么所谓“模块自我迭代”最终会退化成对整个平台源码的直接修改，既无法隔离风险，也无法形成稳定的生命周期治理。

### 2.2 模块职责单一

模块只负责自己擅长的确定性能力，例如：

- IM 模块：消息接入、消息推送、会话映射
- 定时模块：定时触发、调度管理
- 配置模块：配置存取、配置页面
- 日志模块：日志查询、日志聚合
- 文件模块：文件上传、文件转换

模块不负责跨模块理解，不直接承担复杂任务规划。

### 2.3 核心统一编排

模块之间原则上不直接耦合业务逻辑。任何需要跨模块协同的流程，由核心平台统一编排。

### 2.4 统一工作区，运行隔离

所有模块可以位于同一个平台工作区下，便于：

- 智能体统一阅读代码
- 统一生成 Skill
- 统一管理模块清单
- 统一开发和集成

但运行数据必须隔离，避免模块之间互相污染。

进一步说，统一工作区不等于单体运行。工作区可以统一，应用边界必须独立。每个模块都应被视为可独立构建、独立启动、独立部署、独立观测的应用单元。

## 3. 平台分层

### 3.1 核心智能层

职责：

- 用户需求理解
- Skill 读取与选择
- 模块能力路由
- 新模块开发与修复
- 多模块任务编排
- 上下文管理
- 任务拆解与结果汇总

这一层是平台的大脑，是唯一默认拥有智能体能力的部分。

在实现上，这部分能力归属于核心模块，而不是分散注入到各个外挂模块中。核心模块因此既是一个独立应用，又是平台中唯一默认拥有智能体能力的应用。

### 3.2 核心运行时层

职责：

- 模块注册与发现
- 模块启动、停止、重启、守护
- 模块健康检查
- 模块消息路由
- 配置分发
- 权限边界控制
- 日志与观测
- 状态面板与控制接口

这一层不负责业务理解，只负责平台治理。

### 3.3 外挂模块层

职责：

- 提供单一闭环能力
- 按标准协议被核心调用
- 通过统一协议向核心发起请求
- 提供自己的管理说明和 Skill 文档

外挂模块默认不需要理解全局业务，也不需要知道其他模块的内部实现。

但它们必须在工程和运行时层面保持“应用级独立”，即每个外挂模块都应具备自己的入口、生命周期、配置、日志和数据边界，以便被核心独立治理和独立迭代。

## 4. 与当前 MVP 的映射

当前 MVP 已具备若干可直接复用的平台底座：

- 核心编排循环：MVP/src/AgentRuntime.ts
- 统一 HTTP 与 IM 入口：MVP/src/HttpApiServer.ts
- 会话模型：MVP/src/types.ts 中的 AgentSession
- 队列任务模型：MVP/src/types.ts 中的 SessionTask
- 定时任务模型：MVP/src/types.ts 中的 ScheduleTask
- 推送模型：MVP/src/types.ts 中的 PushMessage
- 统一运行时存储目录：MVP/src/Storage.ts

这些能力说明平台已经有核心编排、异步任务、定时触发、消息推送和状态持久化基础。后续模块化演进应优先复用这些概念，而不是并行创造另一套术语。

需要特别注意：这里的“复用”指的是复用概念、模型和必要实现，不指在 MVP 内继续演进代码。凡是进入模块化平台的新实现，都应在 Modules 目录内创建自己的副本、抽象或重写版本。

## 5. 核心平台的定位

核心平台包含两部分：

- 智能体能力
- 平台运行时能力

它既是理解入口，也是模块的统一控制面。核心平台需要长期维护以下全局能力：

- Skill Registry
- Module Registry
- Message Bus
- Config Registry
- Task Orchestrator
- Module Supervisor
- Observability Center

### 5.1 Skill Registry

存放给智能体看的能力说明。智能体通过 Skill 知道：

- 模块能做什么
- 何时调用
- 如何调用
- 参数是什么
- 调用后的结果如何解释
- 模块如何启停或排障

### 5.2 Module Registry

存放给平台运行时看的模块注册信息。运行时通过 Registry 知道：

- 模块入口
- 生命周期策略
- 是否守护
- 是否随核心启动
- 健康检查方式
- 提供哪些动作
- 依赖哪些配置
- 数据目录在哪里

### 5.3 Message Bus

负责模块间统一通讯。所有模块请求、回调、异步通知都通过统一消息协议完成。

### 5.4 Config Registry

负责统一注册和读取模块配置。模块不直接各自读散乱文件，而是通过统一配置协议获取配置。

### 5.5 Task Orchestrator

负责跨模块任务流转。例如：

- IM 收到需求
- 核心理解后调用定时模块
- 定时模块到点后回调核心
- 核心完成后再调用 IM 模块回推用户

### 5.6 Module Supervisor

负责模块进程生命周期和故障恢复。模块崩溃不能拖垮核心平台。

### 5.7 Observability Center

负责日志、状态、心跳、指标、调用链追踪和错误回放。

## 6. 模块分类

### 6.1 接入型模块

作用：连接外部世界。

典型模块：

- IM
- 邮件
- Webhook
- 企业微信
- 飞书
- Telegram
- 浏览器面板

特点：

- 接收外部事件
- 将请求送交核心
- 将核心结果回传外部
- 通常不承担复杂理解逻辑

### 6.2 执行型模块

作用：提供确定性能力。

典型模块：

- 定时调度
- 文件转换
- OCR
- 爬虫
- 数据同步
- 报表生成
- 第三方 API 执行器

特点：

- 输入固定
- 输出固定
- 不承担业务规划
- 可以被核心重复复用

### 6.3 管理型模块

作用：增强平台治理和可操作性。

典型模块：

- 配置中心
- 模块管理面板
- 日志查看器
- 任务监控面板
- 模块市场

特点：

- 偏平台治理能力
- 可能有前端页面
- 对核心透明暴露标准接口

## 7. Skill 与 Manifest 的职责分离

这是平台能否持续演进的关键。

### 7.1 Skill

Skill 面向核心智能体，它解决的是“智能体知不知道怎么用这个模块”。

Skill 需要描述：

- 模块用途
- 适用场景
- 调用动作
- 参数说明
- 返回结果
- 典型示例
- 错误处理建议
- 模块启停说明
- 与其他模块的协作方式

### 7.2 Manifest

Manifest 面向核心运行时，它解决的是“平台知不知道怎么管理这个模块”。

Manifest 需要描述：

- 模块标识
- 版本
- 入口脚本
- 启动命令
- 构建方式
- 健康检查
- 生命周期策略
- 自动启动策略
- 守护策略
- 对外暴露动作
- 依赖配置项
- 数据目录
- 日志目录
- 权限边界

### 7.3 设计原则

Skill 不负责进程管理。Manifest 不负责业务使用说明。两者必须分开维护。

## 8. 通讯模型

模块之间不应直接传业务耦合数据，而应通过统一消息信封通信。

建议统一消息结构至少包含：

- messageId
- traceId
- fromModule
- toModule
- action
- payload
- replyTo
- callbackTopic
- timeoutMs
- context
- createdAt

### 8.1 字段含义

messageId：本次消息唯一标识。

traceId：同一业务链路的全局追踪标识。

fromModule：发起模块。

toModule：目标模块。

action：动作名，例如 create_schedule、execute_task、push_message。

payload：动作参数。

replyTo：处理完成后回给哪个模块。

callbackTopic：异步结果通知通道。

timeoutMs：超时时间。

context：上下文信息，例如 sessionId、claudeSessionId、conversationId、userId、originRequestId。其中 sessionId 用于业务链路关联，claudeSessionId 才用于恢复 Claude 多轮对话。

createdAt：消息创建时间。

## 9. 回调原则

所有模块只需要知道“结果回给谁”，不需要知道完整业务链路。

例如：

- IM 模块不需要知道定时模块内部如何管理触发器
- 定时模块不需要知道核心最终是不是要推送到 IM
- 核心模块不需要知道 IM 内部如何展示消息

每个模块只关心：

- 我收到什么
- 我要做什么
- 做完回给谁

这保证了模块之间的低耦合。

## 10. 典型流程

### 10.1 IM 创建提醒任务

流程如下：

1. 用户在 IM 中发送消息：明天下午三点提醒我交周报。
2. IM 模块将原始消息交给核心。
3. 核心智能体理解意图，判断这是创建提醒任务。
4. 核心调用定时模块的 create_schedule 动作。
5. 定时模块返回创建成功。
6. 核心再调用 IM 模块进行确认回复。
7. 到了触发时间，定时模块回调核心执行提醒任务。
8. 核心完成处理后调用 IM 模块推送提醒结果。

在这个链路中：

- IM 只负责接入和输出
- 定时模块只负责触发
- 核心负责理解、编排和最终结果处理

### 10.2 动态新增模块

流程如下：

1. 用户提出新能力需求。
2. 核心智能体判断当前平台没有该能力。
3. 核心智能体生成新模块开发任务。
4. 核心完成模块代码、Manifest、Skill、测试脚本生成。
5. 核心运行测试。
6. 测试通过后注册模块。
7. 核心运行时启动新模块。
8. 新模块进入平台可用状态。
9. 后续核心智能体可通过 Skill 调用该模块。

### 10.3 模块崩溃恢复

流程如下：

1. 某模块进程崩溃。
2. 核心运行时检测到模块失联。
3. Supervisor 根据策略进行重启或熔断。
4. 记录错误日志与崩溃原因。
5. 核心智能体可在需要时读取日志进行修复。
6. 修复后重新部署模块。
7. 核心平台持续可用。

核心平台必须保证：外挂模块故障不影响核心智能体继续运行。

## 11. 生命周期管理

建议统一模块状态机：

- registered
- installing
- stopped
- starting
- running
- unhealthy
- restarting
- paused
- failed
- disabled

### 11.1 状态说明

registered：已注册，但还未启动。

installing：正在安装依赖或初始化资源。

stopped：已停止。

starting：正在启动。

running：正常运行。

unhealthy：健康检查失败，但尚未完全退出。

restarting：正在重启。

paused：暂时停用。

failed：启动失败或持续异常。

disabled：管理员手动禁用。

这里的生命周期管理对象，应明确是“应用级模块实例”，而不是某段被动加载的源码。只有把模块当成独立应用，registered、starting、running、failed 这类状态才有一致且可治理的含义。

## 12. 配置管理方案

所有模块配置统一注册，不建议各自私有定义且完全自由读取。

建议配置分为三层：

### 12.1 平台全局配置

例如：

- 默认模型
- 默认日志级别
- 消息总线地址
- 核心工作目录
- 全局超时策略

### 12.2 模块配置

例如：

- IM 的 token
- 定时模块的扫描间隔
- Webhook 模块的签名密钥
- 文件模块的大小限制

### 12.3 敏感配置

例如：

- API Key
- 密码
- 第三方密钥

敏感配置应与普通配置分离存储，并在接口层做权限控制。

### 12.4 配置访问原则

模块不直接依赖别的模块的配置文件。模块通过统一配置接口读取配置。配置中心即使未来是一个外挂模块，也必须暴露稳定接口给核心和其他模块使用。

## 13. 数据隔离方案

虽然平台使用统一工作区，但运行数据必须隔离。

建议遵循以下规则：

- 模块源码可在统一工作区中协作
- 模块构建产物应按模块独立管理
- 模块数据必须放在各自私有目录
- 模块日志必须放在各自私有目录
- 模块不得直接写其他模块的数据目录
- 模块间共享只能通过统一通讯机制完成
- 核心维护统一注册信息与全局观测信息

这样既保留统一开发体验，也保留运行隔离性。

除运行数据外，代码演进也应隔离：

- MVP 代码作为冻结样板保留
- Modules 下的核心、接入型模块、执行型模块和管理型模块分别维护自己的源码目录
- 如果某个模块起源于 MVP 中的能力，应在 Modules 下复制出独立版本后再继续维护

## 14. 权限边界

要支持自我迭代，就必须限制智能体和模块的写入边界。

建议：

- 核心智能体拥有模块开发区的受控写权限
- 模块默认只允许写自己的私有目录
- 涉及核心代码变更的操作必须经过显式策略控制
- 模块不得直接修改其他模块运行状态
- 核心统一控制模块注册、上下线和升级

否则后期平台会失控，模块之间会相互污染。

其中有一个关键前提：核心能够被授权修改某个模块，不等于核心可以绕过模块边界把所有能力都写进自己体内。正确的自我迭代方式应当是核心面向目标模块的独立应用边界进行开发、测试、发布和替换。

## 15. 观测与运维能力

平台至少需要统一提供以下观测能力：

- 模块运行状态
- 模块进程 PID
- 启动时间
- 最后心跳时间
- 最近错误
- 最近重启次数
- 调用链 traceId
- 请求成功率
- 配置版本
- 当前启用的 Skill 版本

这些能力建议由核心平台统一聚合展示。

## 16. 最小可落地实施顺序

建议按以下顺序推进：

### 第一阶段

建立模块 Manifest 规范。目标：让核心先能管理模块生命周期。

同时在 Modules 下建立新的核心平台骨架目录，不改动 MVP。

### 第二阶段

建立统一消息信封和回调机制。目标：让模块不再通过业务耦合方式互调。

### 第三阶段

建立 Skill Registry。目标：让核心智能体知道如何调用模块。

### 第四阶段

建立统一配置注册与读取接口。目标：让配置收敛到统一管理模型。

### 第五阶段

接入模块控制台和观测能力。目标：支持启停、重启、查看健康状态和错误日志。

### 第六阶段

开放自我迭代链路。目标：让核心智能体能够开发、测试、注册、部署新模块。

## 17. 代码组织边界

后续实现建议采用如下代码组织方式：

- MVP/ 继续保留为原型参考区
- Platform/ 作为新的平台运行时实现目录，对应一个独立启动的总线与控制面应用
- Modules/platform-core/ 作为智能体核心模块目录，对应一个具备智能体能力的独立应用
- Modules/<moduleId>/ 作为各个外挂模块的独立实现目录，每个目录对应一个独立应用
- Modules/shared/ 作为仅供 Modules 体系内部复用的共享库目录

需要强调：Modules/shared/ 只能承载可复用的共享库，不能替代模块应用本身。一个真正可自我迭代的模块，必须在自己的目录下拥有可独立运行和可独立发布的应用骨架，而不是只在 shared 中拼装依赖后被核心内嵌调用。

其中需要明确新的分层边界：

1. Platform/ 负责平台治理、模块生命周期、消息总线、控制面。
2. Modules/platform-core/ 负责智能体能力、Claude 调用、会话与消息回复。
3. 其他 Modules/<moduleId>/ 负责各自单一能力闭环。

如果需要从 MVP 借用会话、队列、定时、HTTP 或存储相关能力，应采用以下顺序：

1. 先识别需要复用的最小能力边界。
2. 将相关实现复制到 Modules/core 或 Modules/shared。
3. 在 Modules 体系内完成重构、裁剪和扩展。
4. 保持 MVP 原文件不变。

## 18. 当前范围与后续实现边界

本草案定义的是平台契约，而不是现状声明。以下能力属于后续实现阶段，当前 MVP 尚未完整具备：

- 动态模块扫描与注册
- 模块级进程守护与自动重启
- 统一消息总线
- 配置中心 UI 与配置注册表
- 模块级权限控制
- 统一 traceId 追踪

后续实现时，应尽量复用 MVP 已有的 SessionTask、ScheduleTask、PushMessage、Storage 锁模型和 AgentRuntime 编排循环。

## 19. 结论

本平台的核心思想应当明确为：

- 核心模块本身是一个独立应用，并且拥有统一智能体能力
- 核心平台负责全局理解和跨模块编排
- 每个外挂模块都是可独立构建、启动、部署和替换的应用
- 外挂模块只承担单一职责能力
- 外挂模块默认不具备全局智能
- 模块之间通过统一消息协议协作
- 模块配置统一注册管理
- 模块代码可共享工作区，但应用边界、运行数据与生命周期必须隔离
- 平台必须保证模块崩溃不影响核心继续运行
- Skill 负责让智能体会用模块
- Manifest 负责让平台会管模块

这套设计可以同时支持：动态新增模块、模块热接入、统一管理与观测、跨模块任务编排、模块崩溃隔离、模块自修复与自迭代。
